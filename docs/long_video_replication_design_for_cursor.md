# 30 秒以上短视频复刻软件：面向 Cursor 的重新设计文档

> 这不是一份“按既有仓库补丁式扩展”的实施说明，而是一份**从零到一重新设计短视频复刻软件**的产品与系统设计文档。
>
> 目标不是让 Cursor 继承任何现成代码结构，而是让 Cursor 基于这里给出的设计思想、约束判断、能力边界和演进路径，结合它对工程实现的理解，自主设计一套更合理的短视频复刻软件。
>
> 因此，本文刻意减少对既有仓库类名、文件路径、接口细节的绑定，不要求复刻旧实现；更重要的是把问题建模清楚、把系统分层清楚、把 MVP 与长期路线拆清楚。

---

## 1. 文档目标

本文希望 Cursor 最终设计并落地的是一套面向电商、种草、商品展示、口播带货等场景的**短视频复刻软件**，它应支持：

1. 输入一个参考视频，以及补充素材（商品图、参考图、脚本约束、品牌语气、字幕语言、旁白需求等）。
2. 输出一条**风格、结构、节奏、卖点组织方式接近参考视频**，但素材、文案、镜头生成过程可控的新视频。
3. 支持 **30 秒以上** 的成片生成，而不是只覆盖单次 4~8 秒视频生成。
4. 支持后续扩展为批量生产、多版本 AB 测试、平台适配、多语言、多商品 SKU 复用。

这套系统不应被理解为“单模型调用封装器”，而应是一个围绕**视频理解、结构化编排、分段生成、质量校验、拼接导出**建立起来的产品级生成系统。

---

## 2. 重新设计时的总原则

Cursor 在重设计时应优先遵守以下原则。

### 2.1 不继承旧代码结构，继承的是问题理解

应继承的不是某个已有仓库的目录结构、表结构或调度方式，而是以下问题判断：

- 30 秒以上的视频通常不能依赖一次性生成完成。
- “复刻”不等于逐帧临摹，而是复刻其**结构、节奏、叙事方式、卖点铺排、镜头类型与风格约束**。
- 参考视频必须先被结构化，再进入生成阶段。
- 一个可用系统必须把“生成”拆成多个受控阶段，而不是直接把所有输入扔给一个大模型。

### 2.2 结构先行，而不是镜头级全自动幻想

系统的核心不是“尽可能自动分析一切”，而是先得到一个足够稳定的**视频蓝图（blueprint）**：

- 视频想讲什么。
- 目标受众是谁。
- 每一段承担什么转化目标。
- 每一段时长多少。
- 每一段由什么镜头类型组成。
- 哪些段落使用商品特写、模特演绎、对比展示、使用场景、字幕强化、旁白强化。

如果没有蓝图，30 秒以上视频的质量会快速崩坏。

### 2.3 长视频必须 segment-first

对于 30 秒以上成片，系统必须默认采用：

- 先做段落规划
- 再做 segment 规划
- 再逐 segment 生成
- 最后做 merge / 音频对齐 / 字幕合成 / 导出

换句话说，**长视频不是“大视频生成”，而是“多段短视频生成 + 拼接编排”**。

### 2.4 把“可控性”置于“全自动”之前

MVP 应优先保证：

- 用户能看懂系统打算怎么生成
- 用户能修改生成计划
- 用户能替换素材与文案
- 用户能重试某一个 segment
- 用户能只重做脚本、旁白、字幕、某一段视频，而不是整个项目重来

而不是追求一次点按钮就产出完美成片。

### 2.5 先做产品闭环，再做底层完美抽象

优先级应是：

1. 能复刻一条可用视频
2. 能解释为什么这样复刻
3. 能局部修改和重试
4. 能规模化批量生产
5. 再考虑更抽象的通用编排内核

不要在第一阶段就过度设计通用 DAG 引擎、超复杂工作流 DSL 或全能多租户平台。

## 3. 产品定位

这是一个“短视频复刻系统”，不是通用视频剪辑器，也不是单点 AI 视频生成工具。

### 3.1 它解决的问题

给定一条表现不错的参考视频，用户往往想要：

- 换一个商品继续做类似风格的视频
- 同一个商品换多个版本测试投放效果
- 换脚本、换人群、换卖点，但保留原参考视频的节奏与结构
- 把人工拆解 + AI 生成 + 剪辑拼接流程产品化

### 3.2 它的核心价值

系统的核心价值不在于“生成某个片段”，而在于：

- 帮用户把参考视频拆成可复用的结构资产
- 把商品素材与脚本约束映射进结构模板
- 把多模型能力组织成稳定流水线
- 把结果做成可重试、可回放、可解释、可批量化的生产系统

---

## 4. 适用场景

优先服务如下场景：

1. **电商带货视频复刻**
   - 商品展示
   - 卖点轮播
   - 使用场景演示
   - before / after 对比

2. **种草短视频复刻**
   - 情绪开场
   - 痛点切入
   - 体验描述
   - 结果展示

3. **品牌口播/旁白型视频**
   - 脚本强约束
   - 旁白驱动节奏
   - 字幕与卡点重要

4. **矩阵化批量生产**
   - 多 SKU 共用一个结构模板
   - 多语言版本
   - 多封面、多钩子、多结尾版本

---

## 5. 设计范围

### 5.1 MVP 范围

MVP 建议只覆盖以下闭环：

- 导入参考视频与参考图片/商品图片
- 结构化解析参考视频
- 生成一个可编辑的 video blueprint
- 基于 blueprint 自动生成 segment plan
- 逐 segment 生成视频片段
- 生成旁白/字幕/背景音建议
- 合成导出完整视频
- 提供 segment 级重试
- 提供基础质量检查与失败回退

### 5.2 暂不优先范围

第一阶段不建议优先做：

- 镜头级自动精确追踪与严格镜头重建
- 复杂团队协作权限系统
- 极度通用的工作流编排器
- 复杂计费/结算体系
- 大规模素材版权审计系统
- 高级视频编辑时间线 UI

## 6. 用户输入与输出

### 6.1 输入

建议系统支持以下输入对象：

#### A. 参考视频
用于提取：

- 节奏
- 结构
- 段落划分
- 视觉风格
- 镜头类型分布
- 字幕密度
- 旁白风格
- CTA 方式

#### B. 商品素材
包括但不限于：

- 商品主图
- 细节图
- 模特图
- 场景图
- 产品卖点文档
- 规格参数
- 品牌禁用词/必须词

#### C. 文案与策略约束
包括：

- 目标平台（抖音、快手、小红书、视频号等）
- 目标人群
- 主卖点/次卖点
- 口播风格
- 禁止表达
- 时长要求
- 语言版本
- 输出比例（9:16 / 1:1 / 16:9）

#### D. 生成策略偏好
例如：

- 更像参考视频，还是更像新创作
- 更偏商品特写，还是更偏人物演绎
- 是否一定生成旁白
- 是否一定生成字幕
- 是否允许混用静态图动画化与原生视频生成

### 6.2 输出

系统至少应输出：

1. 最终成片视频
2. blueprint（结构蓝图）
3. segment plan（片段生成计划）
4. 每个 segment 的生成结果与元数据
5. 旁白脚本
6. 字幕文件
7. 质量评估报告
8. 可复用模板/项目快照

## 7. 核心设计理念：从“参考视频”到“蓝图”再到“成片”

推荐把系统抽象为三层：

### 7.1 理解层（Understanding Layer）

负责把参考视频与输入素材转化为结构化信息。

典型产物：

- 视频摘要
- 段落切分结果
- 镜头类型标签
- 视觉风格标签
- 字幕/旁白转写
- 商品卖点候选
- CTA 模式

### 7.2 规划层（Planning Layer）

负责把理解结果转化为**可执行生成计划**。

典型产物：

- blueprint
- sequence / chapter plan
- segment plan
- audio / subtitle / merge plan
- retry policy
- quality rubric

### 7.3 执行层（Execution Layer）

负责按照计划调用模型和媒体处理组件完成生成。

典型动作：

- 生成图片
- 生成视频片段
- 生成旁白
- 生成字幕
- 对齐音视频
- 合并导出
- 质检

这三层必须显式分开。不要把“理解 + 规划 + 生成”揉成一个黑盒接口。

## 8. 建议的系统架构

推荐 Cursor 将系统拆成以下模块，而不是围绕某个旧仓库已有模块名去实现。

### 8.1 Project Workspace

项目工作区，管理一次复刻任务的全部输入、状态和产物。

职责：

- 管理原始输入文件
- 管理用户配置
- 记录版本
- 记录每轮生成快照
- 支持回滚与复制项目

### 8.2 Asset Ingestion

素材接入层。

职责：

- 上传与归档素材
- 提取视频基础元信息
- 提取图片尺寸/比例/颜色信息
- 做文件去重与格式规范化
- 为后续理解阶段提供稳定素材引用

### 8.3 Reference Analyzer

参考视频理解层。

职责：

- 视频切段
- 音频转写
- OCR 字幕抽取
- 画面标签识别
- 商品/人物/场景粗分类
- 节奏分析
- 开场钩子识别
- CTA 识别

注意：这层不需要追求完美 CV 精度，重点是为 blueprint 提供足够好的结构线索。

### 8.4 Blueprint Builder

结构蓝图生成器，是整套系统的核心。

职责：

- 基于参考视频理解结果构建视频结构草案
- 结合商品卖点与用户约束重写为目标视频蓝图
- 输出 sequence/chapter/segment 层级结构
- 显式定义每段目标、时长、镜头类型、文案要点和素材来源偏好

### 8.5 Script & Voice Planner

脚本与声音规划层。

职责：

- 为每段生成旁白候选
- 生成字幕文本
- 控制口播节奏
- 控制每段字数与时长匹配
- 输出 TTS plan 或真人录音占位方案

### 8.6 Segment Planner

片段生成规划器。

职责：

- 把 blueprint 中每个段落进一步拆成可执行 segment
- 对齐底层视频生成模型的时长约束
- 为每个 segment 生成 prompt / reference / duration / motion / shot-type / asset binding
- 明确 segment 与最终 merge 的关系

### 8.7 Generation Orchestrator

生成编排器。

职责：

- 调度 segment 生成
- 追踪状态
- 控制并发与重试
- 执行失败恢复
- 管理多模型调用
- 在局部失败时只回滚局部

这层可以是任务队列、工作流引擎或自定义状态机，但对外暴露的概念应保持简单。

### 8.8 Media Processing Pipeline

媒体处理流水线。

职责：

- 视频片段拼接
- 音频混流
- 字幕烧录
- 封面导出
- 分辨率/比例转换
- 码率与格式标准化

### 8.9 Quality Evaluation

质量评估层。

职责：

- 检查时长是否达标
- 检查是否缺段
- 检查字幕与旁白是否错位
- 检查是否出现明显空镜/黑帧/无音频
- 检查与 blueprint 的偏差程度
- 生成通过/重试建议

### 8.10 Template Library

模板资产库。

职责：

- 复用视频结构模板
- 复用镜头模式
- 复用品牌语气模板
- 复用 CTA 模板
- 做批量生产的结构基底

## 10. Blueprint 设计建议

Blueprint 是本系统最重要的数据结构之一。

### 10.1 Blueprint 不应只是脚本文本

它应至少包含：

- 总目标
- 目标受众
- 平台信息
- 视频总时长
- sequence 列表
- 每个 sequence 的任务
- 每个 sequence 的主卖点/辅卖点
- 视觉风格要求
- 旁白策略
- 字幕策略
- CTA 位置与方式

### 10.2 推荐的 Blueprint 层级

建议层级如下：

- Video Blueprint
  - Sequence / Chapter
    - Segment
      - Shot Instruction / Asset Binding / Prompt Pack

这比仅靠线性脚本更稳定，因为它天然支持：

- 局部重试
- 局部换素材
- 多版本变体
- 后续 merge

### 10.3 Blueprint 来源

Blueprint 可以来自三种方式：

1. 纯自动生成：参考视频解析后自动生成草案
2. 人机协同生成：系统先给草案，用户编辑确认
3. 模板驱动生成：从已有行业模板套用后再结合商品重写

MVP 建议采用第 2 种，即**系统生成草案 + 用户可编辑确认**。

## 11. Segment 策略：30 秒以上视频的核心

这是整套设计最重要的部分。

### 11.1 为什么必须 segment 化

因为大多数当前视频生成模型在单次生成时长、稳定性、人物一致性、镜头复杂度上都有明显限制。即使某些模型号称可生成更长视频，在可控性和可重试性上也通常不足以直接承担 30 秒以上成片生产。

因此系统必须把长视频拆成多个生成片段。

### 11.2 Segment 的拆分原则

建议按以下规则拆分：

1. **叙事完整性优先**
   - 一个 segment 最好承载一个小的表达单元
   - 不要跨多个强转折点

2. **时长约束优先**
   - segment 时长必须对齐底层模型支持的可生成范围
   - 如果底层模型只支持离散时长，应在规划阶段做归一化

3. **镜头类型一致性优先**
   - 一个 segment 内尽量保持单一镜头意图
   - 避免又想产品旋转特写、又想人物试背、又想场景切换

4. **参考素材可绑定优先**
   - 一个 segment 应尽量能明确绑定主要参考图/商品图/场景图

5. **可重试性优先**
   - 某个 segment 失败时应只重做它自己，不影响全片其他部分

### 11.3 Segment 的推荐时长策略

Cursor 在实现时应把底层模型时长约束视为一等公民。

如果视频模型只能较稳定地产出 4 / 6 / 8 秒等离散时长，那么规划层应：

- 先根据 blueprint 给出理想时长
- 再把理想时长映射为模型可执行时长
- 必要时拆成多个 segment
- merge 时再通过转场、字幕、音频补齐整体节奏

也就是说，**时长约束不应在执行时临时修补，而应在 segment planning 阶段显式吸收**。

### 11.4 Segment 结果不应只是一段视频

每个 segment 除了生成视频本身，还应产出：

- 使用了哪些 reference
- 用了什么 prompt
- 目标时长和实际时长
- 所属 sequence
- 对应旁白文本
- 对应字幕片段
- 质量评估结果
- 是否建议重试

## 12. 参考视频解析策略

系统不应承诺“精准复刻每个镜头”，而应优先做到“解析出可迁移的结构信息”。

### 12.1 需要重点解析的信息

建议优先关注：

- 开头 3 秒钩子形式
- 段落切换点
- 字幕节奏
- 旁白节奏
- 主要卖点排序
- 商品出镜方式
- 人物出镜比例
- 强 CTA 段的位置
- 镜头密度与切换频率

### 12.2 不必在 MVP 里追求的解析能力

不建议第一阶段强求：

- 极精细镜头边界检测
- 人物骨骼级动作复制
- 精准景深/镜头参数反推
- 完整商业级 CV 标注体系

### 12.3 参考视频解析的最终目标

不是为了“描述原视频”，而是为了生成：

- 一个足够合理的 blueprint 草案
- 一个可执行的 sequence 规划
- 一个可运行的 segment plan

---

## 13. 脚本、旁白、字幕设计

### 13.1 脚本不应独立存在

脚本应与 sequence / segment 绑定，而不是单独生成一大段文案。

每个 sequence 至少应有：

- 主表达目标
- 旁白文本
- 字幕文本
- 关键词
- 禁用词/必含词

### 13.2 旁白规划

建议系统支持三种模式：

1. 无旁白，仅字幕与音乐
2. AI TTS 旁白
3. 用户自定义真人旁白音频

MVP 优先做前两种。

### 13.3 字幕策略

字幕至少应支持：

- 自动切句
- 每段时间轴对齐
- 高亮关键词
- 不同平台的字数密度控制
- 烧录版与外挂版双输出

### 13.4 语言与风格控制

脚本系统应支持：

- 平台语气差异
- 品牌语气约束
- 年轻化/理性型/夸张型/专业型风格切换
- 多语言扩展

---

## 14. 生成执行策略

### 14.1 一次完整生成的建议工作流

推荐工作流如下：

1. 创建项目
2. 上传参考视频与商品素材
3. 解析参考视频
4. 生成 blueprint 草案
5. 用户确认/编辑 blueprint
6. 生成 segment plan
7. 生成脚本/字幕/旁白 plan
8. 逐 segment 执行视频生成
9. 评估每个 segment 质量
10. 对失败或低质量 segment 重试
11. 合并视频、音频、字幕
12. 输出成片与报告

### 14.2 执行层必须支持局部重试

系统至少要支持以下重试粒度：

- 只重做某个 segment 视频
- 只重做某段旁白
- 只重做字幕
- 只重新 merge
- 只替换某段素材后重做该段

### 14.3 执行层必须可观测

无论底层用什么调度方式，都应对外暴露：

- 当前运行到哪一步
- 哪个 segment 失败
- 失败原因是什么
- 是否已自动重试
- 还有哪些产物已可预览

---

## 15. 多模型与 Provider 抽象

这套系统应假设底层会接入多个模型能力，而不是只绑定一个视频模型。

### 15.1 能力类型

至少可以抽象为：

- image generation
- video generation
- speech / TTS
- transcription / ASR
- OCR
- captioning / tagging
- LLM planning / rewrite
- merge / edit / transcode

### 15.2 Provider 抽象原则

Provider 层建议遵守：

- 上层看到的是统一能力接口，不直接感知各家 API 差异
- 不同 provider 的时长、分辨率、输入约束显式声明
- provider 能力差异在 planning 阶段就要被吸收，而不是执行失败后才发现
- provider 返回的结果必须标准化，便于重试、质检、合并

### 15.3 关于底层视频生成器的定位

无论底层接的是 Relay、Google、其他视频模型或后续私有能力，都只应被视为**segment 级视频生成执行器**之一。

它不是整套“长视频复刻方案”本身。

---

## 16. 质量评估设计

没有质量评估，系统就无法进入生产。

### 16.1 至少要做的检查

建议至少包括：

- 时长是否接近目标
- 是否缺少 segment
- 是否生成空文件/黑帧/无音频
- 旁白是否覆盖对应时间段
- 字幕是否越界或错位
- 画面是否偏离该段目标
- 商品是否充分露出
- CTA 是否出现

### 16.2 评估结果不应只有分数

应包含：

- 通过 / 警告 / 失败
- 失败原因分类
- 建议如何重试
- 是否只需重做某段
- 是否允许用户带警告导出

### 16.3 人审插口

MVP 可以把“最终发布前人工确认”作为重要环节。

系统应允许：

- 人工查看 blueprint
- 人工查看 segment 预览
- 人工批准后再最终导出

---

## 17. 用户界面建议

即使第一阶段界面简单，也建议围绕“项目 → 蓝图 → 片段 → 导出”的心智设计，而不是围绕底层任务列表设计。

### 17.1 推荐的主界面信息架构

1. 项目列表
2. 项目详情页
3. 输入素材面板
4. 参考视频解析结果面板
5. blueprint 编辑面板
6. segment plan 面板
7. 生成执行状态面板
8. segment 预览与重试面板
9. 导出面板

### 17.2 用户真正关心的信息

用户通常不关心具体哪个 worker 在跑，而更关心：

- 这条视频的结构是什么
- 每段为什么这么设计
- 哪一段失败了
- 哪一段效果不好
- 怎么改一段而不是推翻全部

### 18.2 素材相关

## 19. 存储与版本策略

### 19.1 需要版本化的对象

建议对以下对象版本化：

- blueprint
- segment plan
- script plan
- voice plan
- merge plan
- 最终导出结果

### 19.2 为什么要版本化

因为短视频复刻天然是高迭代过程：

- 同一个参考视频会有多个重写版本
- 同一个商品会有多个卖点版本
- 同一个 blueprint 会衍生多个 segment 方案

没有版本管理，后期会非常难追溯。

### 19.3 存储对象分类

建议至少区分：

- 原始输入素材
- 解析中间结果
- 规划结果
- segment 生成结果
- merge 结果
- 导出结果
- 评估报告

---

## 20. 失败恢复与重试策略

### 20.1 失败分类

建议至少区分：

1. 输入问题
   - 素材损坏
   - 时长不合法
   - 缺关键素材

2. 规划问题
   - blueprint 不可执行
   - segment 拆分过碎或过长
   - 脚本与时长严重不匹配

3. 生成问题
   - provider 超时
   - provider 失败
   - 返回空结果
   - 结果不满足格式要求

4. 合成问题
   - 音视频长度不匹配
   - 字幕时间轴错位
   - merge 失败

### 20.2 重试原则

- 输入问题应阻断并提示用户修正
- 规划问题优先回到规划层修正
- 生成问题优先只重试单 segment
- 合成问题优先只重做 merge / subtitle / audio align

### 20.3 自动与手动重试结合

建议：

- provider 短暂失败可自动重试
- 质量不佳但技术成功的 segment 允许人工判断后手动重试
- 多次失败后建议用户改 prompt、改 reference、改时长，而不是无限重试

## 21. MVP 实施建议

如果让 Cursor 从零设计并实现，建议按以下顺序推进。

### Phase 1：跑通最小闭环

目标：证明“参考视频 → 蓝图 → 分段生成 → 拼接导出”成立。

建议能力：

- 项目创建
- 素材上传
- 参考视频解析基础版

- 自动生成 segment plan
- 单一视频 provider 完成 segment 生成
- 简单 merge
- 基础导出

### Phase 2：提升可控性

目标：让系统能被真实使用，而不是一次性 demo。

建议能力：

- blueprint 编辑
- segment 手动调整
- segment 级重试
- 字幕与旁白计划
- 质量评估
- 项目版本管理

### Phase 3：提升生产性

目标：支持团队与批量生产。

建议能力：

- 模板库
- 批量生成
- 多 provider 路由
- 多语言版本
- 多平台导出
- 更细粒度的质检与报表

### Phase 4：提升智能化

目标：从“可控生成”走向“更强自动化”。

建议能力：

- 更准确的参考视频理解
- 更智能的 blueprint 重写

## 24. 验收标准建议

系统验收不应只看“是否生成出视频”，而要看是否具备可运营性。

### 24.1 MVP 验收标准

至少满足：

1. 可以导入参考视频和商品素材
2. 可以自动生成一个可读的 blueprint
3. 可以自动生成可执行的 segment plan
4. 可以生成 30 秒以上成片
5. 成片由多个 segment 合成，而不是伪长视频
6. 可以对单个 segment 重试
7. 可以导出最终视频与字幕
8. 可以查看失败原因和基础评估结果

### 24.2 可用性验收标准

至少满足：

- 用户能理解系统生成逻辑
- 用户能修改关键结构
- 用户不用每次都整片重跑
- 同类商品可以复用模板快速生产

### 24.3 工程性验收标准

至少满足：

- 状态可追踪
- 任务可恢复
- 产物可回溯
- 版本可管理
- provider 可替换

## 25. 风险与约束

### 25.1 最大技术风险

- 参考视频解析结果不稳定，导致 blueprint 质量不稳定
- 底层视频模型长时一致性不足
- segment 之间风格不一致
- 旁白、字幕、视频节奏难对齐
- 用户期待“完全自动高保真复刻”，而系统实际更适合“结构复刻 + 风格迁移”

### 25.2 产品风险

- 如果没有 blueprint 可视化，用户会觉得系统像黑盒
- 如果没有 segment 级重试，成本会过高
- 如果没有模板能力，无法规模化
- 如果没有质量评估，生产落地困难

### 25.3 设计取舍

第一阶段必须接受这些取舍：

- 先追求可控和可解释，不追求极限自动化
- 先追求结构复刻，不追求镜头级一比一复刻
- 先追求 segment 级稳定生成，不追求单次超长生成

## 26. 给 Cursor 的最终实施口径

请把这件事理解为：

> 重新设计一套“参考视频驱动的短视频复刻软件”，其核心不是调用某个视频模型，而是建立一条从参考理解、蓝图规划、segment 生成、音视频合成到质量评估的完整生产链路。

你不需要继承任何现有代码实现细节，但应该继承以下核心判断：

1. **30 秒以上视频必须 segment-first。**
2. **复刻的本质是结构复刻，而不是逐帧模仿。**
3. **blueprint 是整套系统的核心中间层。**
4. **系统必须支持局部重试，而不是整片反复重跑。**
5. **底层视频生成器只是 segment 执行器，不是整套方案本身。**
6. **MVP 应优先做出“可解释、可编辑、可重试、可导出”的闭环。**

如果 Cursor 能基于这些原则，自主产出更合理的架构、数据模型、执行编排和交互方式，那么这份文档就达到了目标。

## 27. 一句话总结

这不是一个“把参考视频丢给模型重做一遍”的工具，而应是一套**以 blueprint 为中枢、以 segment 为执行单元、以 merge 与质检为闭环的短视频复刻生产系统**。
